# Portainer one-click install

Portainer operators paste a stack URL or YAML body into the
*Stacks → Add stack* wizard rather than running
`curl … | bash`. This repo ships a pre-rendered Portainer-friendly
compose pair under [`templates/portainer/`](../templates/portainer/)
that imports cleanly via the web URL form.

## What the templates contain

`templates/portainer/compose.yaml` is the canonical standalone-mode
install rendered by [the installer](../installer/) — same `phpfpm` +
`nginx` + `db` + `init` services, same named volumes, same
healthcheck, same X-Forwarded-Proto trust gate. The differences are
operational:

| Knob | Portainer default | Why |
|---|---|---|
| `--edition` | `full` (Magic Sunday charts bundled) | Most operators landing on a one-click installer want the batteries-included setup. |
| `--proxy` | `standalone` | Portainer is rarely paired with Traefik on the same host; if you have one, edit the compose stack labels post-deploy or render your own. |
| `--port` | `28080` | Matches the `curl … \| bash` default. |
| `ENFORCE_HTTPS` | `TRUE` | Portainer typically lives behind a TLS reverse proxy. Set `ENFORCE_HTTPS=FALSE` in the env-var pane if you really need plain HTTP — note the variable is consumed by both the `nginx` and `phpfpm` services, so flip it on both or leave it on neither. |
| `--admin-bootstrap` | off | The headless bootstrap is autogenerated-password-only; Portainer's stack UI redacts logs by default, so the operator never sees the password. Skip the bootstrap and let webtrees' upstream setup wizard ask interactively on first request. |

## Importing the stack

1. **Open Portainer → Stacks → Add stack.**
2. **Build method: *Web URL*.**
3. **Compose URL:**
   ```text
   https://raw.githubusercontent.com/magicsunday/webtrees-docker/main/templates/portainer/compose.yaml
   ```
4. Switch the **Environment variables** tab to *Advanced mode*, paste
   the contents of [`.env.example`](../templates/portainer/.env.example),
   then save.
5. **Deploy.** Portainer pulls the images and starts the stack. First
   request to `http://<your-host>:28080/` lands you in webtrees'
   upstream setup wizard.

The first `compose up` walks the wizard through DB initialisation;
subsequent runs short-circuit because of the marker-file gates in
[`docker-entrypoint.sh`](../rootfs/docker-entrypoint.sh).

### If you hit an HTTPS redirect loop

The X-Forwarded-Proto trust gate
(`rootfs/etc/nginx/includes/trust-proxy-map.conf`) accepts
`X-Forwarded-Proto: https` only from `127.0.0.0/8`, `::1/128`,
`172.16.0.0/12`, and `fc00::/7` by default. Portainer stacks frequently
get attached to operator-defined networks outside that range (custom
IPAM with a 10.x subnet, a pre-created network on the host, an
existing Traefik network on a non-default subnet). When the upstream
TLS proxy lives on such a network, nginx ignores its
`X-Forwarded-Proto: https` header, the enforce-HTTPS 301 fires for
every request, and the browser loops on `ERR_TOO_MANY_REDIRECTS`.

Add the proxy's CIDR to `NGINX_TRUSTED_PROXIES` in the env-var pane
(see [`customizing.md` → HTTPS trust gate](customizing.md#https-trust-gate)
for the hard rules and refusal classes):

```dotenv
NGINX_TRUSTED_PROXIES=10.42.0.0/16
```

The entrypoint validates the CIDR and rewrites
`/etc/nginx/includes/trust-proxy-extra.conf` on every container
restart; a typo crashes the container with a clear stderr message
rather than silently degrading.

## Updating to a newer release

Portainer's *Stacks → … → Update* button re-pulls the registered
compose URL and re-runs `compose up -d`. Because the rendered
templates pin image versions (not `:latest`), updating is opt-in:
the operator chooses when to bump by re-importing this URL after the
upstream repo's `main` branch has rolled a new release.

## Keeping the templates in sync

The static files are regenerated from the installer's Jinja
templates via `make portainer-templates`. The Portainer pair is a
release-time deliverable — if you're a maintainer cutting a release
and the installer or underlying compose templates changed, run:

```bash
make portainer-templates
git diff templates/portainer/        # review
git add templates/portainer/
git commit -m "Refresh Portainer templates for <release>"
```

The rendered files pin the same image tags the installer would emit
for `--edition full --proxy standalone --port 28080`. Re-rendering
keeps the Portainer-side URL contract honest across releases.
